# План своей интеграции ЭДО ↔ iiko (вместо платного коннектора)

Цель: перестать платить 4–6 тыс. ₽/мес за интеграцию партнёра, сделав свой мост **API оператора ЭДО + API iiko**.

## Что обычно делает такая интеграция

Типовая схема:

1. **ЭДО (оператор)**  
   Приём/отправка электронных документов: УПД, ТОРГ-12, акты и т.п. (получение входящих от поставщиков, отправка исходящих).

2. **iiko**  
   Учёт приходов товара: создание/подтверждение документов поставки (incoming supplies), привязка к номенклатуре iiko.

3. **Мост между ними**  
   - Получить документ из ЭДО → разобрать номенклатуру и количества → создать или обновить приход в iiko (или наоборот: приход в iiko → сформировать и отправить документ в ЭДО).  
   - Сопоставление товаров ЭДО ↔ номенклатура iiko (артикулы, названия, единицы).

То есть «под капот» у партнёра именно это: **API оператора ЭДО + API iiko**, плюс маппинг и бизнес-логика.

---

## Что нужно выяснить (у партнёра / из договора / из интерфейса)

Чтобы повторить функционал своими силами, важно зафиксировать:

| Вопрос | Зачем |
|--------|--------|
| Какой именно **оператор ЭДО** используется? (Диадок, СБИС, ЭДО.Поток, Такском, другой) | У каждого свой API и тарифы. |
| Это **iiko Cloud** или **iiko Server** (он-прем)? | Cloud: `api-ru.iiko.services` (REST). Server: свой набор API (в т.ч. документы). |
| Какие **типы документов** идут через интеграцию? (УПД, ТОРГ-12, акты приёма-передачи и т.д.) | Понять объём и форматы на стороне ЭДО. |
| Направление потока: **ЭДО → iiko**, **iiko → ЭДО** или **двусторонний**? | От этого зависит, что именно писать первым. |
| Есть ли **веб-интерфейс** у коннектора (логи, очередь документов, маппинг)? | Помогает понять сценарии и поля без доступа к коду. |

Договор и описание услуги часто содержат название оператора ЭДО и формулировки вида «интеграция с iiko» — этого уже достаточно, чтобы выбрать нужные API.

## Сторона ЭДО: операторы с API

Для разработки своей интеграции подойдут операторы с открытым/документированным API:

- **Контур.Диадок** — [api-docs.diadoc.ru](https://api-docs.diadoc.ru/), HTTP API, SDK (C#, Java, C++).
- **СБИС (Saby)** — [saby.ru/help/integration/api/edo](https://saby.ru/help/integration/api/edo) — ЭДО API.
- **ЭДО.Поток (edo.ru)** — открытый API, роуминг с другими операторами.
- **Такском** и др. — у каждого свой портал для разработчиков.

Тарифы у операторов обычно за «ящик» и кол-во документов; интеграция через API чаще всего входит в стандартный функционал, без отдельной платы 4–6 к за «коннектор».

---

## Сторона iiko: API для документов и поставок

- **iiko Cloud (РФ)**  
  Документация: [api-ru.iiko.services](https://api-ru.iiko.services/docs).  
  Нужно найти разделы, связанные с:
  - документами поставок (incoming supplies / documents),
  - номенклатурой (для маппинга строк из ЭДО в товары iiko).

- **iiko Server (он-прем)**  
  [iikoAPI](https://api.iiko.ru/) — доступ к документам (приходные/расходные накладные, акты и т.д.), номенклатура, отчёты.  
  Детали по методам документов — в официальной документации и при необходимости у api@iiko.ru.

- **EDI API iiko (Resto)**  
  В iikoOffice при старте создаются системы EDI (Внешняя, Внутренняя, **Контур EDI**). Через REST доступны методы: список заказов по поставщику (`GET edi/{ediSystem}/orders/bySeller`), квитирование (`PUT orders/ack`), подтверждение ответа поставщика (`PUT response`), **выполнение заказа / накладная** (`PUT edi/{ediSystem}/invoice` — тело в формате ediMessage XML).  
  Это даёт путь: **УПД из Диадока → преобразование в ediMessage → PUT invoice** в iiko. Подробно: [edo-iiko-iiko-edi-api.md](edo-iiko-iiko-edi-api.md).

Текущий проект (datalens-iiko-etl) уже использует данные iiko (OLAP, проводки, приходы INVOICE) — значит, окружение и доступ к API либо есть, либо их можно добавить для нового сервиса.

## Архитектура своей интеграции (эскиз)

```
[ Оператор ЭДО ]  ←→  [ Ваш сервис-мост ]  ←→  [ iiko API ]
     (API)                  (логика, маппинг)         (документы, номенклатура)
```

Компоненты вашего моста:

1. **Клиент к API оператора ЭДО**  
   Авторизация, получение входящих документов, отправка исходящих, статусы (по документации выбранного оператора).

2. **Клиент к iiko API**  
   Получение номенклатуры, создание/обновление документов поставки (приходов), при необходимости — чтение существующих приходов.

3. **Маппинг**  
   Справочник соответствия: строка документа ЭДО (наименование, код, единица) ↔ номенклатурная позиция iiko (id/артикул). Хранить в БД или конфиге.

4. **Оркестрация**  
   - По расписанию или webhook от ЭДО: «новый входящий документ» → скачать → разобрать → создать/обновить приход в iiko.  
   - При необходимости: по событию из iiko или по расписанию — формировать исходящий документ и отправлять в ЭДО.

5. **Хранение и логи**  
   Чтобы не терять документы и разбирать ошибки: сохранять сырые документы ЭДО, статусы отправки в iiko, ошибки маппинга (например, в БД или файловом хранилище).

Реализация может быть в виде отдельного репозитория или сервиса (например, Python/Node бэкенд + очередь задач), без привязки к текущему ETL инвентаря — ETL остаётся для отчётности, ЭДО-мост — только обмен документами с iiko.

## Практические шаги

1. **Зафиксировать** у партнёра/из договора: оператор ЭДО, тип iiko (Cloud/Server), перечень документов и направление потоков.
2. **Зарегистрироваться** в личном кабинете выбранного оператора ЭДО, открыть доступ к API (ключи, документация).
3. **По документации iiko** найти методы по документам поставок и номенклатуре для вашего варианта (Cloud или Server).
4. **Сделать минимальный прототип**: один сценарий (например, «входящий УПД из ЭДО → создание прихода в iiko»), без сложного маппинга — жёстко зашить 1–2 товара для проверки.
5. **Добавить маппинг и обработку ошибок**, затем второй поток (если нужен) и вывести в прод с мониторингом.
6. **Отключить** платный коннектор после уверенного прогона своей интеграции на тестовых и боевых документах.

---

## Важно с юридической стороны

«Украсть всё» здесь имеется в виду не копирование кода партнёра, а **понимание сценариев и повтор реализации на основе открытых API**. Такой подход легален: вы пользуетесь своими контрактами с оператором ЭДО и iiko и пишете свой код. Договор с партнёром стоит проверить на предмет запрета на «аналогичную» интеграцию после прекращения услуги — обычно такого запрета нет, так как интеграция с ЭДО и iiko стандартна.
