# Разбор интеграции EDI-Doc (партнёр): как устроена

По материалам из двух папок: **воровство** (плагин + логи EDI-Doc 3.02) и **стек2** (набор DLL — «стек» интеграции).

---

## 0. Что в каких папках

| Папка | Содержимое |
|-------|------------|
| **воровство** | Полный набор: все DLL (в т.ч. Resto.Plugin.AlcoholDeclaration, Adapters), подпапка **EDI-Doc 3.02** с дневными логами (.log). По логам восстановлены сценарии и стек вызовов. |
| **стек2** | Только DLL — тот же стек зависимостей без логов и без плагина алкогольной декларации: DevExpress.*, Diadoc.dll, DiadocApi.dll, IikoHiddenApi.dll, Leaf.xNet.dll, LicCryptolens.dll, Logger.dll, RestSharp.dll, SbisApi.dll, ServeApi.dll, System.*. Нет exe, конфигов, логов. |

Вывод: **стек2** — это «оголённый» набор библиотек интеграции (ЭДО + iiko); сценарии и URL/методы восстанавливаются по папке **воровство** (логи + те же DLL).

---

## 1. Что за продукт

- **Название:** EDI-Doc, версия **3.02**.
- **Тип:** плагин под **iiko** (судя по DLL `Resto.Plugin.*` — плагин для iiko Front / рабочего места менеджера).
- **Назначение:** работа с входящими накладными из ЭДО и их связка с iiko (сопоставление товаров, при необходимости — создание/проведение приходов).

---

## 2. Стек технологий (по DLL и логам)

| Компонент | Назначение |
|-----------|------------|
| **Diadoc.dll, DiadocApi.dll** | Работа с **Контур.Диадок** (получение накладных, контрагенты, документы ЭДО). |
| **SbisApi.dll** | Поддержка **СБИС** как второго оператора ЭДО (в логах в данном срезе только Диадок). |
| **IikoHiddenApi.dll** | Обращения к **iiko Server** (скорее всего неофициальные/«скрытые» методы REST). |
| **ServeApi.dll** | Обёртка над REST API iiko Server: в стеке вызовов виден метод **Suppliers_price** (прайс/номенклатура поставщиков). |
| **Leaf.xNet.dll** | HTTP-клиент (запросы к iiko и, при необходимости, к ЭДО). |
| **RestSharp.dll** | Ещё один HTTP-клиент (возможно, для Диадока или других сервисов). |
| **DevExpress.*** | UI (формы, гриды, отчёты). |
| **LicCryptolens.dll** | Защита лицензии (Cryptolens), привязка к машине (id в логах). |
| **Logger.dll** | Запись логов. |

Вывод: интеграция = **API Диадока (и при необходимости СБИС) + REST API iiko Server**; плагин работает внутри iiko и сам ходит к серверу iiko по HTTP.

---

## 3. Подключения (из логов)

### 3.1. iiko Server (REST)

- **URL:** `https://oreks-co.iiko.it:443/resto`  
  Т.е. используется **iiko Server** (он-прем), не Cloud; хост и порт — ваши.
- **Авторизация:** логин + пароль (в логе: `Alexandra_P` и хеш пароля).
- Ошибки в логах: «Ошибка подключения к серверу iiko», «Ошибка обработки накладной» при недоступности этого хоста (SSL/сеть). Обработка накладной идёт с вызовом iiko (см. ниже).

### 3.2. Контур.Диадок

- В логе: «Успешно подключились Контур API-*a540a11a-829c-444b-84f3-16f00513392c* , *manager@pizza-sushi.com* , *yAA-WHA-N2x-mv8*».
- То есть: API key (или идентификатор подключения), логин пользователя Диадока, токен/пароль.
- Перед работой с накладными: «Смена Юр Лица» → «Получаем контрагентов» → «Данные Загружены».

### 3.3. Лицензия

- Cryptolens, идентификатор объекта лицензии: `49023e1b-6e3a-6c33-0133-cce1f6f50009`.

## 4. Сценарий работы (по логам и стеку)

Типичная последовательность:

1. **Старт плагина**  
   Плагин запускается (в контексте iiko Front), пишет «Плагин Запустился версия 3.02».

2. **Подключения**  
   - Чтение url + login + pass (iiko) → запросы к `.../resto`.  
   - Проверка лицензии Cryptolens.  
   - Подключение к Диадоку (Контур API + пользователь + токен).  
   - Загрузка данных: смена юр. лица, контрагенты.

3. **Работа с накладными**  
   - Пользователь открывает вкладку плагина («Нажали на нашу вкладку»).  
   - «Получаем накладные» — список входящих накладных из ЭДО (Диадок).  
   - Пользователь выбирает накладную («Переходим к накладной &lt;номер&gt;», например Г0000015853, 4825, 01834).

4. **Обработка одной накладной (загрузка данных из iiko)**  
   При открытии накладной плагин:
   - Загружает документ ЭДО (XML УПД/счёт-фактура и т.п.).  
   - Вызывает **ServerApi.Suppliers_price(Employee, Start)** → обращение к iiko Server (REST).  
   Судя по названию, это загрузка прайса/номенклатуры поставщиков из iiko для подстановки в форму и сопоставления строк накладной с товарами iiko.
   - При недоступности iiko в лог попадает «Ошибка обработки накладной» и стек с `Server.ServerApi.Suppliers_price` → `PageFirst.LoadData` → `LoadDocumentЭДО`.

5. **Сопоставление и сохранение**  
   - Пользователь сопоставляет строки накладной ЭДО с номенклатурой iiko.  
   - «Сохранение сопоставлений» — сохранение маппинга (куда именно — в конфиг/БД плагина или в iiko — по логам не видно).

6. **Документ ЭДО**  
   В логах при ошибке выводится фрагмент XML документа ЭДО: УПД (Универсальный передаточный документ), формат 5.03, КНД 1115131, таблица товаров (наименование, кол-во, цена, сумма и т.д.). Этого достаточно, чтобы со своей стороны парсить такие же УПД из Диадока и формировать из них приходы.

Итого: интеграция работает по цепочке **ЭДО (Диадок/СБИС) → плагин (сопоставление) → iiko Server REST**. Точное создание прихода в iiko (какой метод REST вызывается после «Сохранение сопоставлений») по логам не видно — возможно, отдельное действие пользователя или фоновый вызов после сохранения маппинга.

## 4.1. Что нашли в DLL через ILSpy (как появляется вкладка «EDI-Doc 3.02»)

Ключевое: **пункт меню/вкладка создаётся самим плагином внутри DLL**, а не через внешний конфиг iikoOffice.

В `Diadoc.dll` найден класс:

- **`DiadocNS.ITSPlugin : INavBarPlugin, IPlugin`**
  - `MenuName = "EDI-Doc 3.02"`, `Version = "3.02"`.
  - `MenuGroup` возвращает группу с двумя пунктами (две вкладки):
    - `TabPageFirst`
    - `TabPageSecond`
  - Путь к папке плагина/логам: `AppDomain.CurrentDomain.BaseDirectory + "Plugins\\\\" + MenuName + "\\\\"` → ожидается `...\\Office\\Plugins\\EDI-Doc 3.02\\`.

UI сделан на DevExpress:

- `PageFirst : XtraUserControl` — основной экран, табы **Черновики / Контрагенты / Входящие / Накладная**.
- Настройки берутся из `Settings.Default.SettingsMain` (JSON `ConfigCL`), сохраняются через `Settings.Default.Save()`.

TabPages (каркас навигации) в DLL:

- `TabPageFirst : PluginTabPageBase`, `Name = "Документы"`:
  - `CreateControl()` → `new PageFirst()`, `new PageFirstController(pageFirst)`
  - `LoadData()` → `controller.OnLoadData()`
  - `GetTabPageId()` → `"Документы"`
- `TabPageSecond : PluginTabPageBase`, `Name = "Настройки"`:
  - `CreateControl()` → `new Настройки()`, `new PageSecondController(settings)`
  - `LoadData()` → `controller.OnLoadData()`
  - `GetTabPageId()` → `"Настройки"`

Подключения:

- iiko: из `RestApiClient.CurrentSessionAuthData` (serverUrl/login + PasswordHash/PasswordSha1Hash), далее `ServerApi` и `IikoHiddenApi`.
- ЭДО: `DiadocMain(ApiToken, Login, Password)` или СБИС (`API_SBIS(Login, Password)`).

Создание накладной в iiko:

- `CreateiikoInvoice()` формирует `Download_inv_Document` и вызывает `ServerApi.Download_Invoice(...)` (в логе/коде парсится XML `documentValidationResult/documentNumber`).
- Сопоставления и связь адрес→склад сохраняются через `IikoHiddenApi.SaveOrUpdateAnnouncements(...)` (JSON `CompareNews`/`CompareiikoPos`).

## 5. Выводы для своей интеграции

- **ЭДО:** используется **Контур.Диадок** (и заложена поддержка **СБИС**). Документы — УПД в XML (формат ФНС/Диадок).
- **iiko:** только **iiko Server**, не Cloud. Доступ по **REST** к базовому URL вида `https://&lt;хост&gt;:443/resto`, авторизация логин/пароль.
- **Ключевой метод iiko в логах:** `Suppliers_price` — получение данных для сопоставления (прайс/номенклатура поставщиков). Остальные вызовы к iiko (создание прихода и т.д.) нужно либо восстанавливать по другим логам/действиям пользователя, либо находить в документации/разборе iiko Server API.
- **Сопоставление:** строки накладной ЭДО ↔ номенклатура iiko; маппинг сохраняется и используется при «обработке» накладной.
- **Реализация партнёра:** плагин к iiko Front (C#, .NET), DevExpress UI, лицензия Cryptolens. Для своей замены можно не делать плагин, а сделать отдельный сервис (например, веб или консоль), который будет дергать **Диадок API** и **REST iiko Server** (в т.ч. по аналогии с `Suppliers_price` и при необходимости — методы документов поставок/приходов, если они есть в API).

Дальше имеет смысл:  
1) зафиксировать точный список REST-вызовов к iiko (хост, путь, параметры), которые делает плагин при «Получаем накладные», «Suppliers_price» и после «Сохранение сопоставлений»;  
2) по документации iiko Server / Resto API или по перехвату трафика восстановить недостающие методы для создания приходов;  
3) реализовать свой сервис: Диадок API (получение УПД, контрагенты) + REST iiko (номенклатура/прайс, приходы) + хранение и применение сопоставлений.
